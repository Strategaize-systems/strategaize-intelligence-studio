## Reihenfolge-Logik

Die Signalkette (SLC-001–005) bildet das Fundament:
1. **Setup** (SLC-001) — Infrastruktur, DB, Layout
2. **Inbox** (SLC-002) — Rohsignale aufnehmen
3. **Analyzer** (SLC-003) — Signale klassifizieren → Insights
4. **Patterns** (SLC-004) — Insights zu Mustern verdichten
5. **Improvements** (SLC-005) — Verbesserungen ableiten

Die Bewertungskette (SLC-006) bewertet Chancen strukturiert.

Die Entscheidungskette (SLC-007) erzwingt Entscheidungen und erzeugt Action Triggers.

Die Produktionskette (SLC-008–013) produziert Assets, Module, Wissen und strukturiert Tests:
- **SLC-008 Brand** vor **SLC-009 Content** (Brand-Kontext fließt in Generierung)
- **SLC-010 Experiments** und **SLC-011 Research** hängen von Opportunities ab
- **SLC-012 Module Drafts** hängen von Patterns + Opportunities ab
- **SLC-013 Knowledge** hängt von Insights + Patterns + Improvements ab

Die Integration (SLC-014) vernetzt alles:
- Dashboard mit Statistiken
- Cross-Objekt-Verlinkungen
- FTS-Suche

## Parallelisierungsmöglichkeiten

Folgende Slices können parallel bearbeitet werden:
- **SLC-004 + SLC-006** (beide hängen von SLC-003 ab, nicht voneinander)
- **SLC-008** (hängt nur von SLC-001 ab, kann parallel zu SLC-002–007)
- **SLC-010 + SLC-011 + SLC-012** (alle hängen von SLC-006 ab, nicht voneinander)

## Key Design Notes

1. **SLC-008 (Brand) vor SLC-009 (Content)**: Brand-Konfiguration muss existieren bevor Assets generiert werden, weil die Brand Config als Kontext in jeden Generierungs-Prompt fließt.
2. **Action Triggers als eigene Entität**: SLC-007 erzeugt Action Triggers als First-Class-Objekte in eigener Tabelle — nicht als Feld am Quell-Objekt. Das ermöglicht eigenständige Nachverfolgung und Status-Workflow.
3. **SLC-014 als letzter Slice**: Dashboard und Integration können erst sinnvoll gebaut werden, wenn alle Module existieren. Die Platzhalter-Seite aus SLC-001 wird dann ersetzt.
