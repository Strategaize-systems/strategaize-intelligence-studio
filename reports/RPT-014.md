## Findings

### Medium

#### M1 — `saveBrandProfile` ist nicht atomar (UPDATE + Changelog-INSERT separat)
**Location:** [src/server/brand/repository.ts:122-155](../src/server/brand/repository.ts#L122-L155)

Der Update-Pfad fuehrt zwei separate Supabase-Calls aus:
1. `UPDATE brand_profile SET version=newVersion, data=...`
2. `INSERT INTO brand_profile_changelog (...)` mit den Diff-Eintraegen

Falls Step 2 fehlschlaegt (Connection-Drop, Constraint-Violation, RLS-Block), ist das Profile bereits auf der neuen Version, aber **ohne** zugehoerigen Changelog-Eintrag. Der Audit-Trail wird inkonsistent — Versions-Sprung ohne Diff-Spur.

**Impact:** V1-relevant nur theoretisch. Single-Tenant + Internal-Tool + niedriger Traffic. Reale Failure-Wahrscheinlichkeit = sehr niedrig. Audit-Vollstaendigkeit aber im Versprechen der Slice-Spec ("Versions-Snapshot pro Aenderung mit Changelog-Audit").

**Akzeptabel fuer V1:** Ja. Die Anwendung ist single-User, wahrscheinlichste Failure-Mode (Validation-Error) wird vor jedem DB-Touch abgefangen. Falls Changelog-INSERT trotzdem failt, wirft die Action `Error("saveBrandProfile changelog: ...")` und der User sieht eine Fehlermeldung — kein silent-fail.

**Empfohlene Mitigation (V1.1+):** Entweder
- (a) RPC-Funktion `save_brand_profile_atomic(jsonb, uuid)` in PL/pgSQL mit Transaktion, oder
- (b) DB-Trigger `brand_profile_changelog_on_update` der den Changelog automatisch schreibt (Profile-UPDATE als single Operation, Trigger ergaenzt automatisch).

**Gehe als ISSUE-008 in `KNOWN_ISSUES.md`.**

### Low

#### L1 — API-Inkonsistenz: `saveBrandProfileAction` returns Result, `getActiveBrandProfileAction` throws
**Location:**
- [src/server/brand/actions.ts:29-55](../src/server/brand/actions.ts#L29-L55) — return-Pattern
- [src/server/brand/actions.ts:61-72](../src/server/brand/actions.ts#L61-L72) — throw-Pattern

`saveBrandProfileAction` liefert `SaveActionResult` mit `{ success, error, fieldErrors }` — ideal fuer Client-Components mit `try {} catch {}` oder Result-Discrimination.

`getActiveBrandProfileAction` wirft `Error("Nicht eingeloggt")` bei fehlendem Auth — RSC kann das via Error Boundary fangen, Client-Component braucht aber `try { ... } catch (e) { ... }` und Type-Inference verliert das Result-Pattern.

**Impact:** UI-Pattern-Frage fuer /frontend SLC-102 MT-3. Komponente muss zwei verschiedene Aufruf-Patterns implementieren, was Konsistenz erschwert.

**Empfehlung:** In /frontend SLC-102 entscheiden, ob `getActiveBrandProfileAction` auf Result-Pattern umgestellt wird (Konsistenz) oder ob beide Server-Actions throw werden (RSC-Pattern). Beides ist akzeptabel; aktuell mixed.

**Gehe als ISSUE-009 in `KNOWN_ISSUES.md`.**

#### L2 — `BrandProfileValidationError.fieldErrors` ohne aggregierten Sektion-Summary
**Location:** [src/server/brand/repository.ts:45-52, 86-94](../src/server/brand/repository.ts#L45-L94)

`fieldErrors` ist `Record<string, string[]>` mit Zod-Issue-Pfaden (z.B. `"sections.painPoints.costs.money": ["Pflichtfeld"]`). Fuer React-Hook-Form `setError` per Pfad ideal. Fuer Toast-Anzeige ("3 Sektionen unvollstaendig: Pain Points, Brand Voice, Goals") muss die UI selbst aufloesen via `SECTION_LABEL_DE`.

**Impact:** Keine. UI-Verantwortung in /frontend MT-3. Backend liefert genug Information, UI muss aggregieren.

**Empfehlung:** In /frontend SLC-102 MT-3 einen Helper `summarizeSectionErrors(fieldErrors): SectionKey[]` bereitstellen und im Toast die deutschen Labels nutzen. Kein Backend-Change noetig.

**Gehe als ISSUE-010 in `KNOWN_ISSUES.md` (UI-side Mitigation in SLC-102 Frontend).**
